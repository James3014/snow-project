# ADR-0001: 微服務架構採用

## 狀態
已接受

## 背景
原有單體架構存在職責混合、難以擴展、技術債務累積等問題。

## 決策
採用微服務架構，將系統拆分為：
- user-core: 用戶核心服務
- resort-services: 雪場資訊服務  
- snowbuddy-matching: 雪伴匹配服務
- calendar-service: 行事曆服務
- gear-service: 裝備管理服務
- social-service: 社交功能服務

## 理由
1. 單一職責原則
2. 獨立部署和擴展
3. 技術棧靈活性
4. 團隊自主性

## 後果
### 正面
- 服務邊界清晰
- 可獨立開發部署
- 故障隔離

### 負面  
- 分散式系統複雜性
- 網路延遲
- 數據一致性挑戰
